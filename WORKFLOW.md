# 工作流约定

> 本文件用于固定约束“生产目录”和“预发布目录”的职责边界，避免以后推送 GitHub 时混用目录。

---

## 1. 目录职责

### 生产目录

- 路径：`/opt/openai-cpa`
- 作用：当前正在运行的正式环境
- 要求：
  - **继续保留原上游 `wenfxl/openai-cpa` 的更新提醒**
  - 方便第一时间知道原作者什么时候更新
  - 不作为“对外分享版本”的直接推送来源

### 预发布目录

- 路径：`/home/ubuntu/openai-cpa-prepush`
- 作用：准备推送到 `BFanSYe/GPT-Auto-Manager` 的固定工作目录
- 要求：
  - 面向“你自己的 GitHub 分发版本”
  - 支持“跟随你自己的 GitHub Release 的更新提醒”
  - 支持“一键自动更新”
  - 所有准备推送到 GitHub 的更新，都必须先在这里整理、验证，再推送

---

## 2. 更新提醒规则

### 生产目录 / 生产容器

- 保持指向：`wenfxl/openai-cpa`
- 目的：追踪原作者更新节奏

### 预发布目录

- 指向：`BFanSYe/GPT-Auto-Manager`
- 目的：让使用你仓库的人看到“你的 GitHub”版本提醒

---

## 3. 推送 GitHub 的固定流程

以后凡是涉及以下动作：

- 推送到 GitHub
- 更新 release 分支
- 发新 tag / release
- 整理可公开分享版本

都必须按下面顺序执行：

1. 先把本次改动落到 **预发布目录**：`/home/ubuntu/openai-cpa-prepush`
2. 在 **预发布目录** 做功能验证
3. 验证通过后，再从 **预发布目录** 推送到 GitHub
4. 如有需要，再把验证通过的内容同步到生产目录 / 生产容器

---

## 4. 至少要做的验证

每次准备推送 GitHub 前，至少应根据改动范围完成下列验证中的对应项：

### 基础验证

- Python 语法 / 编译检查
- 关键 API 烟雾验证
- 页面版本号 / 文档版本号检查

### 功能验证

- 注册主流程相关改动：做真实回归或定向验证
- 邮箱后端相关改动：做对应单测 / 实测
- Sub2API / CPA 相关改动：做配置读取与请求链路验证
- 自动更新相关改动：在隔离 smoke 环境做端到端验证

---

## 5. 生产同步规则

只有在以下条件满足时，才允许同步到生产目录 / 生产容器：

- 改动已经先进入预发布目录
- 预发布目录验证通过
- 需要把这版能力实际部署到当前运行环境

---

## 6. 当前固定结论

### 已确定

- 生产目录继续保留“看上游更新提醒”
- 预发布目录负责“面向你自己 GitHub 的版本整理与推送”
- Clash 代理体系继续用我们自己的实现
- 不直接接入上游 `v10.1.4` 的 Clash 部署管理

### 以后默认行为

如果用户说“推 GitHub / 发版本 / 整理可分享版本”，默认理解为：

- **先操作预发布目录**
- **先验证**
- **再推送**

而不是直接从生产目录推送。
